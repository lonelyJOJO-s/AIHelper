## 核心知识

1. 负载均衡的目标不只是平均分配请求，而是在实例能力、实时负载、延迟和可用性约束下合理调度流量。
2. Round Robin 只关注请求数量，无法感知慢请求堆积；P2C（Power of Two Choices）随机选择两个节点，再结合负载、流量或延迟择优，能以较低成本避开繁忙节点。
3. 健康检查应区分 Liveness、Readiness 和深度监控。非核心依赖故障不应轻易让实例退出流量池，否则可能引发节点批量摘除和故障放大。
4. Full GC 会触发 STW，使应用线程暂停。负载均衡器通常通过连续超时、延迟升高、活跃请求堆积和成功率下降间接识别异常节点；摘除与恢复应使用连续阈值、迟滞机制和慢启动。
5. 应用扩容会同步放大数据库连接、Redis 连接、下游 RPC 和消息队列压力。例如 100 个实例、每实例 100 个数据库连接，理论上可产生 10,000 个连接并压垮数据库。
6. 重试必须设置次数和总时间预算，只对可恢复且安全的错误重试，并使用换节点、退避、随机抖动和重试预算，避免重试风暴。
7. 熔断不是单个请求重试耗尽后立即触发，而是依据统计窗口内的失败率、慢调用比例和最小请求数，在 Closed、Open、Half-Open 状态间转换。
8. 创建订单等有副作用的接口只有在下游具备幂等能力时才适合重试。重试必须复用同一个幂等键，并由唯一约束、幂等记录或状态机保证结果一致。
9. Redis 锁适合快速去重和降低并发竞争，但不能单独承担最终正确性；MySQL 唯一索引或同库事务通常作为最终兜底。
10. `finally` 只能处理仍能正常执行代码流的释放锁场景，无法覆盖 kill -9、机器宕机或 JVM 崩溃。Redis 锁应具有租约，并通过唯一 token 和 Lua 原子校验后删除，防止误删其他请求的锁。
11. 跨服务的订单与库存一致性可以选择分布式事务，也可以采用本地事务、可靠消息、幂等消费、状态机和补偿。分布式事务在业务代码表面可能直观，但协调、锁持有、回滚和故障恢复的工程复杂度通常更高。

## 用户观点

- 用户指出，对于固定存在长短请求差异的接口，简单轮询无法真正解决节点负载不均，并结合 go-zero 的 P2C 实现说明：先随机选择两个节点，再依据流量和负载决定最终节点。
- 用户判断，把非相关依赖纳入健康检查会导致实例被摘除，增加其他节点压力。
- 用户识别出应用扩容后 `实例数 × 单实例连接池大小` 可能直接打垮数据库。
- 用户提出重试应有 2～3 次上限，超过整体异常阈值后应熔断。
- 用户认为下游做好幂等后，上游可以对超时请求重试。
- 用户指出真正的难点是保证幂等判断与订单创建的原子性，并质疑 Redis 与 MySQL 之间的一致性问题。
- 用户进一步询问 `finally` 是否能覆盖持锁进程崩溃，以及分布式事务是否会更简单。

## 纠正与澄清的误区

- 慢节点问题不是轮询“分配数量错误”，而是轮询不了解节点实时繁忙程度。
- Full GC 的主要影响不只是 CPU 或内存占用，而是 STW 期间应用线程无法处理请求。
- CPU、内存和流量指标适合监控、扩缩容和权重调整，不适合单独作为硬性的健康摘除条件；低 CPU 的线程死锁实例也可能完全不可服务。
- 单个请求超过重试次数不等于立即熔断整个下游；熔断需要基于统计窗口和阈值。
- “接口支持幂等”仍要求同一业务请求复用同一个幂等键；先查后写无法抵御并发，必须依赖唯一约束、条件更新或原子操作。
- Redis 常用于幂等加速或锁，但 Redis 写入与 MySQL 订单创建无法天然原子提交，因此不能只靠 Redis 保证最终正确性。
- `finally` 无法在进程已崩溃时执行，必须依赖锁过期机制；释放锁时还要校验锁的所有者。
- 分布式事务并不天然比消息和补偿简单，它只是把部分业务状态机复杂度转移到了事务协调与故障恢复机制中。
